# Эмулятор Game Boy на Nokia 9110: первый настоящий кадр

Nokia 9110 Communicator — не самый очевидный аппарат для эмуляции Game Boy.

Среда разработки здесь из другой эпохи: PC/GEOS, Borland C++ 4.52, сегментированная память, movable resources, near- и far-указатели и цепочка сборки, чьи сообщения иногда напоминают археологические находки.

Именно поэтому задача оказалась интересной.

Цель легко сформулировать и трудно подделать: выполнить настоящий ROM Game Boy на реальной Nokia 9110, вывести его экран и управлять игрой с клавиатуры коммуникатора.

## Лестница вместо прыжка

Я не стал сразу заталкивать ядро эмулятора в SDK.

Первые этапы были намеренно скучными:

1. создать приложение GEOS;
2. нарисовать статическую картинку;
3. запустить обновление по таймеру;
4. выделить полноценный framebuffer 160×144;
5. выполнить синтетическую нагрузку, похожую на работу CPU;
6. и только после этого подключать Peanut-GB.

Такая лестница оказалась принципиальной. После каждой ошибки оставался нижний рабочий уровень, к которому можно было вернуться.

## Peanut-GB встречает C из девяностых

Peanut-GB — компактное и быстрое ядро Game Boy, но исходный код рассчитан на гораздо более современную среду C, чем Borland C++ 4.52.

Рабочая адаптация использовала compile-time direct hooks для:

- чтения ROM;
- доступа к cartridge RAM;
- сообщений об ошибках;
- вывода LCD-строк.

Это оказалось не только оптимизацией. Сохранённые в структуре ядра callback-указатели приводили к падению в тестовой среде. Прямые хуки работали стабильно.

## Ловушка 64 КиБ

Первая попытка встроила весь ROM размером 32 КиБ прямо в приложение.

Это было ошибкой.

В 16-битной модели памяти ROM начинал конкурировать с состоянием ядра, framebuffer, стеком и глобальными переменными в ограниченном near-data пространстве. Формально проект собирался, но запуск стал ненадёжным.

Рабочая архитектура загружает ROM из файловой системы GEOS в два отдельно выделенных и заблокированных блока по 16 КиБ.

Позже из фиксированных неинициализированных данных был вынесен и framebuffer.

## Ошибка, которой не было в эмуляторе

Одна из самых странных сборок зависала ещё до появления окна.

Под подозрением были загрузчик ROM, память и ядро.

Причиной оказалась identity приложения.

GEOS использует permanent name и token. Повторное применение старой identity позволило сохранённому состоянию вмешаться в запуск новой диагностики. Свежие имя и token сразу вернули видимый поэтапный старт.

С тех пор новое диагностическое ответвление получает новую identity.

## Первый настоящий кадр

Рабочая диагностика показала:

- ROM загружен;
- `gb_init()` завершился успешно;
- CPU выполнил тысячи шагов;
- PPU вывел 144 строки;
- ошибок ядра нет;
- настоящий гостевой кадр завершён.

Это ещё не была игра. Это было доказательство, что вся цепочка замкнулась:

```text
приложение GEOS
  -> загрузчик файла
  -> память ROM
  -> CPU Peanut-GB
  -> PPU Game Boy
  -> преобразование строк
  -> bitmap GEOS
  -> экран Nokia
```

## От одного кадра к игре

Затем появился осторожный цикл на 10 FPS, потом 30 FPS и схема 60 гостевых кадров при 30 выводимых. В эмуляторе SDK она показала около 59 гостевых FPS.

Управление добавлялось только после стабилизации цикла:

- стрелки — D-pad;
- Space или Z — A;
- X — B;
- Enter — Start;
- Tab — Select.

После этого ROM уже выполнялся и принимал команды.

Оставался главный тест — реальная Nokia.

И реальное железо оказалось куда менее вежливым, чем эмулятор на ПК.
